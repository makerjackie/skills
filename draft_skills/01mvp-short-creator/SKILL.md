# 01MVP 短教程生成器（TLDR）

将用户输入整理为一篇可发布的短教程文档，默认落地到 `content/docs/tldr/`。

## 执行流程
1. 识别主题与目标结果。
2. 判断是否已经给出解决方案，如果还没给出，应该创建文档放到： content/docs/tldr/unsolved
3. 生成文件名、标题、描述。
4. 按双版本结构写入正文。
5. 自检篇幅与改写边界后保存。

## 文件与 Frontmatter
- 输出路径：`content/docs/tldr/<slug>.mdx`。
- `slug` 规则：英文小写 + 连字符，结尾建议加 `-tldr`；若重名追加 `-2`、`-3`。
- frontmatter 至少包含：

```mdx
---
title: <自动生成，一个简短的问题，例如："如何快速实现 MVP"> 
description: <一句话说明这篇 TLDR 解决什么问题>
---
```

## 固定正文结构
正文标题从 `##` 开始，不使用 `#`。

```mdx
## 版本 A：原始输入（轻润色）

<仅修正错别字、断句、标点、换行，不新增观点，不改变原意。>

## 版本 B：优化版

<保持简短，补充必要背景/步骤/注意点；优先补可立刻执行的信息。>
```

## 改写边界（必须遵守）
- 保留用户核心观点与语气，不改立场。
- 版本 A 只做轻整理，尽量保留原句与原顺序。
- 版本 B 可以补充必要信息，但必须是“太长不看”风格。
- 信息不足时明确标注“可选补充”，不要编造事实。
- 默认使用简体中文，采用“先结论，后步骤”的表达。

## 篇幅与排版
- 整篇建议控制在 300-800 中文字。
- 版本 B 不超过版本 A 的 1.5 倍。
- 单段不超过 5 行，优先短段落与列表。
- 命令、路径、配置使用代码块或行内代码。
- 提示块使用 `<Callout type="...">`，不要使用 `:::warning` / `:::tip`。

## 执行时的操作要求
- 完成后直接保存到 `content/docs/tldr/`。
- 用户若明确指定标题、文件名、语气，优先按用户要求。
- 未被明确要求时，不改动页面 UI、通用组件和构建配置。
